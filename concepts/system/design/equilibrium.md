# Equilibrium (system design)

**Equilibrium** means a system has reached a steady balance.

Things are still moving, but the overall state stays roughly the same because opposing forces cancel out.

In plain terms: **inflow and outflow are balanced**.

## System view

- If work arrives as fast as work is completed, the queue stays stable.
- If mistakes are introduced as fast as they are corrected, quality stays flat.
- If maintenance matches drift, structure stays clear.

Equilibrium is not "perfect" or "finished." It is a **current balance point** under current conditions.

If conditions change (load, resources, rules), the balance point shifts.

The same pattern appears across physical, economic, social, and personal systems. Cross-domain examples live under [Equilibrium subconcepts and examples](equilibrium/README.md), using the [Concept - Detail Pages](../../Concepts/ideas/detail.md) split. [Entropy](entropy.md) is the drift that pulls systems away from balance; [proportional allocation](proportional_allocation.md) is one way to split finite totals fairly; the [pipeline](../troubleshooting/pipeline.md) view helps see where inflow and outflow stop matching.

## Quick test

Ask over a short time window:

1. Is the important level mostly stable?
2. Are in and out rates close?
3. If not, which side is stronger?

If levels keep rising or falling, the system is not in equilibrium yet.
